# Phase 6.5: Special Operator Preview

## Summary

Implement special-operator preview as an **app-level, modifier-aware multi-pass render plan**: one "normal" mesh plus optional **overlay meshes** for `#` and `%`, generated by source-to-source transforms and rendered with distinct Three.js materials.

## Effort Estimate

Medium (1–2d) for MVP in current-file scope; Large (3d+) for full fidelity across `use/include` graphs.

## Action Plan

1. **Codify semantics + precedence**: `*` = "not in any pass," `!` = "restrict visible set," `#/%` = "style overlays" (stackable; `*` wins, `!` filters first, then `#` vs `%` style).
2. **Add a modifier-aware parser**: lightweight tokenizer + brace/paren matcher that ignores strings/comments and extracts "object statements" as spans with leading modifiers.
3. **Build an "operator map" + effective-visibility resolution**: walk the statement tree to compute `{disabled, showOnly, highlight, background}` per node with nesting/stacking rules.
4. **Generate per-pass source variants** (string transforms using spans):
   - **Main pass**: strip `#/%/!` tokens; remove `*` nodes; if any `!`, disable everything not under `!`.
   - **`#` overlay pass**: keep program context, disable all non-`#` geometry nodes.
   - **`%` overlay pass**: same as `#`, but for `%`.
5. **Render orchestration + caching**: extend pipeline to run 1–3 renders when needed; cache by `(engine, passKind, transformedSourceHash, renderSettings)`.
6. **Three.js composition + materials**: load each pass as a separate `Group`; apply pass-specific materials, `renderOrder`, and `polygonOffset`/`depthWrite=false`.
7. **Tests + fallback**: unit tests for tokenization/span extraction + pass-source generation; golden scene screenshot tests; fall back to single-pass if parsing/extra passes fail.

## Operator Semantics (Standard OpenSCAD)

- `#` **highlight**: shown prominently in preview (debug emphasis); does not change geometry
- `%` **background/transparent**: shown faintly/transparent; does not change geometry
- `*` **disable**: object is ignored (removed from evaluation)
- `!` **show only**: only `!`-marked objects shown; everything else hidden
- Stacking: modifiers can stack (e.g., `#!`); `*` is absolute removal, `!` filters first, then `#/%` style

## Parsing Strategy

- Minimum viable grammar:
  - Skip `//...` and `/*...*/`, skip strings `"..."` with escapes
  - Track `()[]{}` depth to find statement boundaries
  - Recognize modifier prefix as contiguous `[#%*!]+` tokens before object statement
- Output: tree of nodes `{spanStart, spanEnd, modifiers, children[]}`

## Rendering Approach (Why Multi-Pass)

- STL cannot carry per-part styling; OpenSCAD debug operators are preview/UI concepts
- Multi-pass meshes + compositing is the practical approach

## Per-Operator Behavior

- `#`: generate overlay mesh → highlight material
- `%`: generate overlay mesh → background material
- `*`: exclude from all passes
- `!`: change which nodes are included in main pass (gate overlays to `!`-visible subset only)

## OpenSCAD CLI/WASM Support

- `--colorscheme` mainly useful for image outputs, not STL → Three.js workflow
- CSG exports don't preserve debug operator metadata
- Plan: source transforms + separate exports on both CLI and wasm

## Three.js Composition

- Scene graph: `mainGroup` (normal mesh), `highlightGroup` (higher renderOrder), `backgroundGroup`
- Use `polygonOffset` or `renderOrder` + `depthWrite=false` on transparent overlays to reduce z-fighting

## Material Design

- **Normal**: existing `MeshStandardMaterial`
- **Highlight (`#`)**: magenta/pink, ~35–60% opacity, optional emissive/glow
- **Background (`%`)**: neutral gray, ~5–15% opacity, `NormalBlending`, `depthWrite=false`
- `!`: not a material — an inclusion mode; optionally add UI badge "Show-only active"
- `*`: hidden; optionally render gutter/inline marker in Monaco

## Performance

- Only render overlay passes if operators are present
- Cache outputs per pass by hash; reuse mesh when edits don't affect a pass's transformed source
- Debounce multi-pass renders more aggressively than single-pass

## Edge Cases

- **Nested/stacked modifiers**: `#!%object;` → resolve to one effective style (`#` wins over `%` visually)
- **Boolean operations**: STL-based mesh cannot show subtractive operands; overlays rewrite boolean nodes to `union` in overlay passes (heuristic)
- **Operators in libraries**: MVP only parses current buffer; full correctness requires transforming dependency graph
- **Operators on modules/for/if**: parser must handle these block structures

## Testing Strategy

- Parsing: fixtures covering comments/strings, nested blocks, stacked modifiers, `module` bodies, `if/for` blocks
- Transform correctness: snapshot tests of each generated pass source
- Visual regression: deterministic camera + lighting + renderer screenshot for canonical models

## Fallback Behavior

- If multi-pass fails: render main pass only (current behavior), surface non-blocking warning
